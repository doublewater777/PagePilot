# Commit Convention

PagePilot commit messages are version-scoped. Every commit must declare which app release line it belongs to, and that version must match the canonical source in the repo.

## Canonical version

Two version files serve different roles:

| File | Role |
|------|------|
| `SHIPPED_VERSION` | App Store live version (提审基准) |
| `VERSION` | Current repo / commit version |

```text
SHIPPED_VERSION=1.0.8
VERSION=1.0.8
```

Commit messages must match `VERSION`. 提审默认目标版本是 `SHIPPED_VERSION` 的下一个 patch。

When bumping a release, update `VERSION` first, then sync project templates:

```bash
./scripts/bump-version.sh 1.1.0
```

That script updates:

- `VERSION`
- `Integrations/**/project*.yml`
- local `project.yml` if it exists (generated by `make spm`, gitignored)

After a release commit, tag the version:

```bash
git tag v1.1.0
```

## Message format

```text
<version>: <description>
```

Rules:

1. The subject line must start with `major.minor.patch:` followed by a space and a short description.
2. The version must exactly match `VERSION` in the working tree.
3. If `VERSION` is staged for a bump, the commit may use the new staged version.
4. Use present tense, imperative mood for the description.
5. Keep the subject under 72 characters when practical.

### Examples

```text
1.0.8: feat: improve WatchRemote UI
1.0.8: fix: guard PRO state on foreground
1.0.8: chore: sync App Store metadata for zh-Hans
1.0.8: release
1.1.0: release
```

### Optional type prefix

You may add a Conventional Commits type after the version:

```text
1.0.8: feat(reader): add highlight notes
1.0.8: fix(watch): throttle digital crown commands
```

Supported types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `build`, `release`.

## Exempt messages

These auto-generated subjects skip validation:

- `Merge ...`
- `Revert "..."`
- `fixup! ...` / `squash! ...`

## Enforcement

Install the repo hooks once per clone:

```bash
./scripts/setup-hooks.sh
```

The `commit-msg` hook rejects commits that:

- omit a version prefix
- use a non-semver version
- use a version that does not match `VERSION` (or the staged bump)

## Workflow

### Day-to-day feature work

Work on the current release line. Every commit uses the version in `VERSION`:

```bash
git commit -m "1.0.8: feat: add reading streak badge"
```

### Start a new release line

For App Store submission, use the full prep script instead of a manual bump:

```bash
./scripts/prepare-submission.sh
# update metadata/version/<version>/ whatsNew, then:
./scripts/commit-release.sh
```

See `docs/agents/submission-flow.md` for the full 提审 checklist.

Manual bump only when not submitting:

```bash
./scripts/bump-version.sh 1.1.0
git add VERSION Integrations iPhone/Info.plist WatchRemote/Info.plist
git commit -m "1.1.0: release"
git tag v1.1.0
```

### Hotfix on an older line

Check out the release branch, ensure `VERSION` matches that line, then commit with that version. Do not reuse a newer version number on an older branch.

## Why version-scoped commits

- `git log --grep '^1.0.8:'` shows all work for a release
- changelog and release notes are easier to assemble
- agents and humans can tell whether a change belongs to the current shipping version
- accidental version drift is caught before it lands
